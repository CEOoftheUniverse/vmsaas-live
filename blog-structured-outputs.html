Structured Outputs: How to Get Reliable JSON from LLMs

Unstructured text is fine for a chatbot. For agents that need to feed data into downstream systems, you need reliable, schema-compliant JSON — every time, not 95% of the time. Here are the techniques that actually deliver.

Every agent developer eventually runs into the "almost valid JSON" problem: the model outputs JSON that's 99% correct but missing a closing bracket, or uses a different field name than you specified, or wraps the JSON in a markdown code block. Here's how to make structured output reliable enough for production.

The 4 approaches, from weakest to strongest

1. Prompt-only ("please output JSON")

Ask the model to output JSON in the prompt. Works most of the time with frontier models, fails unpredictably in edge cases. Acceptable for low-stakes prototyping, not for production.

Use when: rapid prototyping, non-critical outputs

2. JSON mode

Most frontier models support a JSON mode flag that guarantees the output is parseable JSON. It does NOT guarantee adherence to your specific schema — just valid JSON syntax. A significant improvement over prompt-only.

Use when: you need valid JSON but can tolerate schema variation

3. Structured outputs with schema (recommended)

OpenAI, Anthropic, and Google all support passing a JSON Schema alongside your prompt. The model is constrained to output JSON matching that schema exactly. Field names, types, and required fields are enforced at the model level.

Use when: production agents that feed structured data downstream

4. Constrained decoding (logit masking)

At the token generation level, only tokens consistent with the schema are allowed. Guarantees schema compliance even for models that don't natively support structured outputs. Used in Outlines, llama.cpp grammar mode, and similar tools.

Use when: open-weight models or zero-tolerance schema compliance

Structured output with Pydantic + MoltBot

from pydantic import BaseModel from moltbot import Agent from typing import List, Optional class ExtractedContact(BaseModel): name: str email: Optional[str] company: str role: Optional[str] confidence: float # 0.0–1.0 agent = Agent(model="claude-opus-4") result = agent.run( prompt="Extract contact info from this email thread...", output_schema=ExtractedContact # MoltBot enforces schema ) # result is a validated ExtractedContact instance — guaranteed print(result.name) # str, always present print(result.email) # Optional[str], None if not found

Schema design tips for reliable outputs

  • Prefer Optional fields over required when uncertain: A required field the model can't find will either be hallucinated or cause a failure. Use Optional and handle None in downstream code.
  • Use enums for categorical fields: Instead of status: str, use status: Literal["active", "inactive", "pending"]. Dramatically reduces hallucinated values.
  • Add descriptions to fields: Pydantic's Field(description="...") gives the model context for ambiguous fields. A field named "confidence" without a description will be interpreted differently by different models.
  • Keep schemas flat when possible: Deeply nested schemas (3+ levels) increase error rates. If you need complex structure, consider multiple chained extractions.
  • Always validate and handle errors: Even with constrained generation, implement a try/catch + retry loop. Budget for 1–2% failure rate in high-volume pipelines.
